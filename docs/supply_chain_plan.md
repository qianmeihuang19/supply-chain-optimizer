# 供应链计划优化 AI 项目 — 完整项目计划 (v3.0)

## 一、项目概述

### 1.1 项目目标
构建一个基于数学优化（Mathematical Optimization）的供应链发货计划与终端调拨优化系统Demo，实现：
- **预测驱动的库存预置**：基于销售预测，提前将货物布局到距终端x天交付距离内，确保订单确认后可在x天内交付
- **发货计划优化（引擎一）**：综合多维参数，输出总成本最优的分批滚动发货计划
- **终端调拨优化（引擎二）**：基于终端剩余库存和周边需求，输出处置成本最优的调拨方案
- **闭环反馈**：通过ETA/ATA对比、预测偏差统计、终端库存滞留等反馈数据持续提升系统预测和优化能力

### 1.2 核心设计理念：可配置的目标交付天数x

系统引入参数x（目标交付天数），定义从客户订单确认到货物送达客户的最大天数。
- x=1：隔天交付（最快）
- x=2：确认后两天交付
- x≥(干线在途+本地配送)：不需要预置库存，纯响应式发货

x可按客户、目的地独立配置。x本质上是控制预置库存深度的旋钮：x越小，预置越深，库存成本越高但服务水平越好；x越大，预置越浅，库存成本低但响应慢。

核心等式：
```
预置提前量(天) = 干线在途时间 - (x - 本地配送时间)
若预置提前量 > 0：需要提前发货，货物须在订单确认前到达或接近终端
若预置提前量 ≤ 0：不需预置，订单确认后发货即可
```

Demo场景（本地配送统一1天）：
| 目的地 | 干线在途(正常) | x=1 需提前 | x=2 需提前 | x=3 需提前 | x≥5 不需预置 |
|--------|-------------|-----------|-----------|-----------|-------------|
| 天津 | 2天 | 2天 | 1天 | 不需要 | — |
| 大连 | 3天 | 3天 | 2天 | 1天 | — |
| 长春 | 4天 | 4天 | 3天 | 2天 | — |

| bias_direction | float | 系统性偏差方向(正=客户习惯多报,负=少报) | +0.15 | 系统计算 |
| bias_correction | float | 偏差修正系数(预测量×此系数=修正后预测) | 0.85 | 系统计算 |
| sample_size | int | 计算所用的历史样本数 | 45 | 系统计算 |
| last_updated | datetime | 最后更新时间 | 2025-06-01 | 系统自动 |

置信度规则：
- 上限0.95（永远保留意外空间），下限0.40（太低失去预置意义）
- Demo阶段：管理员手动设初始值，系统后台积累偏差数据
- 数据足够后（≥30个订单周期）：系统开始给出建议值，人工确认
- 计算方法：指数加权移动平均(EWMA)，近3个月权重高于6个月前
- 分层统计：按客户×目的地×时段×订单规模分层，提升准确性

#### 3.1.8 动态安全库存参数 (safety_stock_params) ★新增★
| 字段 | 类型 | 说明 | 示例 | 来源 |
|------|------|------|------|------|
| param_id | string | 编码 | "SS001" | 系统自动 |
| dest_id | string | 终端编码 | "CC" | 管理员 |
| service_level_z | float | 服务水平系数Z | 1.65 (对应95%) | 管理员 |
| demand_std_sigma | float | 需求波动标准差σ(托盘/天) | 3.0 | 管理员初始设定，系统根据偏差历史自动更新 |
| replenishment_lead_days | float | 补货提前期L(天) | 5.0 | 管理员初始设定，系统根据ETA偏差自动更新 |
| calculated_safety_stock | float | 计算值=Z×σ×√L(托盘) | 11.07 | 系统计算 |
| manual_override | int | 人工覆盖值(NULL=使用计算值) | NULL | 管理员 |
| last_updated | datetime | 最后更新时间 | 2025-06-01 | 系统自动 |

安全库存公式：安全库存 = Z × σ × √L
- Z：服务水平系数，由管理员按客户重要性设定（90%→1.28, 95%→1.65, 99%→2.33）
- σ：需求波动标准差，基于历史预测偏差自动计算并持续更新
- L：补货提前期，= 下单处理时间 + 干线在途时间 + 到货入库时间
- 补货频率越高（L越小），安全库存越低：如L从5天降到2天，安全库存减少36%

| delta_quantity | int | 与原始预测差异(正=多于,负=少于) | 系统计算(confirmed - forecast_qty) |
| delta_vs_adjusted | int | 与修正后预测差异 | 系统计算(confirmed - adjusted_qty) |
| status | string | "pending"(等待确认)/"customer_confirmed"(客户确认)/"arrival_alarm"(到货未确认告警)/"manual_confirmed"(人工确认)/"cancelled"(取消) | 系统自动流转 |
| alarm_triggered_at | datetime | 到货告警触发时间(货物到达时客户未确认则自动填充) | 系统自动(ATA录入时触发) |
| confirmed_notes | string | 人工确认时的备注(系统人员记录决策依据) | 系统人员输入 |

**关键设计原则**：
订单确认与货物到达是两条完全独立的时间线，无先后依赖关系。

**到货即检查机制★核心★**：
货物到达终端的那一刻，系统立即检查该笔预测对应的订单确认状态：
- 已确认(customer_confirmed) → 直接进入汇合流程（差异分析+本地配送）
- 未确认(status仍为pending) → **立即**发起ALARM给系统人员，status变为"arrival_alarm"

**四种确认场景**：

```
场景A - 提前确认(客户确认时间 < 货物到达时间)：
  客户主动提交确认 → confirmed_by="CUST_xxx" → status="customer_confirmed"
  → 货物到达时系统检查：已确认 ✓
  → 直接安排差异分析和本地配送

场景B - 同步确认(客户确认时间 ≈ 货物到达时间)：
  货物到达时客户恰好提交 → confirmed_by="CUST_xxx" → status="customer_confirmed"
  → 正常汇合流程

场景C - 到货时未确认 → 立即告警 → 客户后续自行确认：
  ① 货物到达终端
  ② 系统立即检查：订单未确认 → 发起ALARM
     status从"pending"变为"arrival_alarm"
     alarm_triggered_at = 货物到达时间
  ③ 系统人员收到告警（待处理队列标红）
  ④ 在系统人员介入之前，客户自行提交了确认
     → confirmed_by="CUST_xxx" → status="customer_confirmed"
     → 告警自动解除，进入正常汇合流程

场景D - 到货时未确认 → 立即告警 → 系统人员人工确认：
  ① 货物到达终端
  ② 系统立即检查：订单未确认 → 发起ALARM
     status从"pending"变为"arrival_alarm"
     alarm_triggered_at = 货物到达时间
  ③ 系统人员收到告警（待处理队列标红）
  ④ 系统人员主动处理：
     - 联系客户获取最终需求量
     - 或根据业务判断人工确认(预测值或调整值)
  ⑤ 在系统中执行人工确认：
     confirmed_by="SYS_USER_xxx" → status="manual_confirmed"
     confirmed_notes="决策依据"
  ⑥ 系统自动进入汇合流程（差异分析+本地配送）
```

**汇合逻辑**：
系统在以下条件满足时触发本地配送：
1. 货物已到达终端（或在途且预计x天内到达）
2. 订单确认已完成（status="customer_confirmed"或"manual_confirmed"）

## 四、优化模型设计

### 4.0 双引擎架构总览

| 维度 | 引擎一：发货计划优化 | 引擎二：终端调拨优化 |
|------|-------------------|-------------------|
| 触发时机 | 货物出发前（基于预测提前布局） | 货物到达终端且订单确认后（处置剩余） |
| 核心决策 | 发多少、何时发、用哪辆车、预置多深 | 剩余库存调拨/退货/存储如何最经济 |
| 决策变量 | 发货量、车辆分配、发货时间、滚动批次 | 调拨量、退货量、存储量 |
| KPI | 总成本最小化(运费+违约+存储+预置持有) | 处置成本最小化(调拨+退货+存储+货损) |
| 关键输入 | 销售预测、置信度、x值、运力、运价、安全库存 | 剩余库存、未交付需求、调拨运价、需求概率 |
| 输出反馈 | 发货计划→资源锁定→运单 | 滞留库存→引擎一下批次扣减 |
| 不确定性来源 | 预测准确度(用置信度量化)、运力可用性 | 订单确认偏差(已确定) |

两个引擎共享OR-Tools求解框架和基础数据层，模型定义独立。引擎二输出(终端滞留库存)反馈引擎一，形成闭环。

### 4.1 引擎一：发货计划优化模型

#### 4.1.1 决策变量
```
x_ship[i,v,t] = 分配给目的地i、使用车辆v、在时间t发货的托盘数量（整数）
y_use[i,v,t]  = 是否使用车辆v在时间t向目的地i发货（0/1二元变量）
```

#### 4.1.2 目标函数（最小化总成本）
```
Minimize:
  ── 运输成本 ──
  Σ freight_cost(x_ship[i,v,t], pricing_mode[v], unit_price[v,i])
  
  ── 违约成本(延迟交付转换为成本) ──
  + Σ penalty_cost(arrival_date[i,t] + local_delivery[i] - required_date[i], penalty_rule[customer])
  
  ── 终端存储成本(预置库存的持有成本) ──
  + Σ storage_rate[i] × preposition_stock[i] × preposition_days[i]
  
  ── 资金占用成本(预置库存的资金时间价值) ──
  + Σ cargo_value × time_decay_rate × preposition_stock[i] × preposition_days[i]
  
  ── 安全库存持有成本 ──
  + Σ storage_rate[i] × safety_stock[i] × holding_days[i]
  
  ── 装载率激励(装载率越高,单位成本越低) ──
  - Σ loading_rate_bonus(x_ship[i,v,t], capacity[v])
```

#### 4.1.3 约束条件
```
1. 库存就绪约束(核心-与x联动) ★新增★：
   terminal_stock[i,d] + arriving_in_transit[i, d..d+(x-local_delivery)] 
   >= expected_demand[i,d] × confidence[customer,i]
   （订单预计确认日，终端库存+x天内到达的在途≥修正后预期需求）

2. 需求满足约束：
   Σ_v,t x_ship[i,v,t] >= adjusted_demand[i]
   （总发货量≥置信度修正后的预测需求+安全库存）

3. 车辆容量约束(体积)：
   x_ship[i,v,t] × pallet_volume <= vehicle_volume × y_use[i,v,t]

4. 车辆容量约束(重量)：
   x_ship[i,v,t] × pallet_weight <= vehicle_max_weight × y_use[i,v,t]

5. 车辆可用性约束：
   y_use[i,v,t] <= available[v,t]

6. 运力匹配约束：
   Σ_i y_use[i,v,t] <= 1（每辆车每时段只能去一个目的地）

7. 终端存储约束：
   terminal_stock[i] + arriving[i] <= storage_capacity[i]

8. 在途库存约束：
   在途数量纳入可用库存计算（"距终端x天内"视为可用）

9. 安全库存约束 ★新增★：
   terminal_stock[i] >= safety_stock[i] = Z[i] × σ[i] × √L[i]
   （终端库存不低于动态安全库存水位）

10. 终端滞留反馈约束：
    下批发货量扣减终端滞留库存

11. 滚动补货约束 ★新增★：
    单批预置量 <= adjusted_demand × max_preposition_ratio
    （不一次性推全部预测量，分批滚动补货）
```

#### 4.1.5 滚动补货策略

系统不一次性将全部预测量推送到终端，而是分批滚动补货：

```
批次1(高置信度部分): 提前(在途时间+缓冲)天发货
  发货量 = 预测量 × 置信度 × 第一批比例(如60%)
  
批次2(验证后追加): 提前(在途时间)天发货
  发货量 = 根据前几天订单确认趋势验证预测准确度后决定
  
批次3(响应式补货): 订单确认后发货(如果x足够大)
  发货量 = 实际确认量 - 已预置量
  
紧急补货: 终端库存低于安全水位时触发
  使用加急运输，成本为正常的1.5-2倍
```

补货频率最优平衡点：
- 高频小批 → 安全库存低(√L小)、库存持有成本低，但单位运输成本高(装载率低)
- 低频大批 → 安全库存高(√L大)、库存持有成本高，但单位运输成本低(装载率高)
- 最优点 = 总成本(运输+库存持有)最低处，由引擎一自动求解
- UI提供敏感性分析：用户拖动补货频率滑块，实时显示总成本曲线

### 4.2 装载率优化子模型

```
输入：
  - 托盘尺寸 (长×宽×高, 重量)
  - 可叠放层数
  - 车辆内部尺寸 (长×宽×限高)
  - 车辆限重
  - 装载间隙要求

输出：
  - 最大装载托盘数
  - 装载率(%) = 实际装载数 / 理论最大装载数
  - 装载方案示意图（Demo展示用）

约束：
  - 体积约束：托盘总体积 ≤ 车厢容积（考虑间隙）
  - 重量约束：托盘总重量 ≤ 车辆限重
  - 高度约束：叠放后总高度 ≤ 道路限高要求
  - 取较低值：min(体积可装数, 重量可装数)
```

### 4.3 引擎二：终端调拨优化模型

#### 4.3.1 决策变量
```
t[i,j] = 从终端i调拨到终端j的托盘数量（整数）
r[i]   = 从终端i退回发货点（上海）的托盘数量（整数）
s[i]   = 终端i保留存储的托盘数量（整数）
b[i,j] = 是否使用回程运力从终端i调拨到终端j（0/1二元变量）
```

#### 4.3.2 目标函数（最小化总处置成本）
```
Minimize:
  ── 调拨运输成本 ──
  Σ_i,j t[i,j] × 调拨单价[i,j] × (1 + 二次调拨惩罚系数 × is_secondary[i,j])
  
  ── 退货运输成本 ──
  + Σ_i r[i] × 退货运价[i→上海] × (1 - 回程折扣率[i] × b_return[i])
  
  ── 退货处置成本 ──
  + Σ_i r[i] × 退货处置费用[i]
  
  ── 继续存储成本 ──
  + Σ_i s[i] × 存储单价[i] × 预计存储天数[i]
  
  ── 货损成本 ──
  + Σ_i,j t[i,j] × 货值 × 货损概率
  + Σ_i r[i] × 货值 × 货损概率
  
  ── 资金占用时间衰减成本 ──
  + Σ_i s[i] × 货值 × 日衰减率 × 预计存储天数[i]
  
  ── 调拨收益(负成本：避免目标终端违约) ──
  - Σ_i,j t[i,j] × 目标终端避免的违约成本[j]
```

#### 4.3.3 约束条件
```
1. 库存平衡约束：
   surplus[i] = Σ_j t[i,j] + r[i] + s[i]

2. 需求匹配约束：
   Σ_i t[i,j] <= unfulfilled_demand[j]

3. 存储容量约束：
   current_stock[j] + Σ_i t[i,j] <= capacity[j]

4. 调拨时间窗约束(与x联动) ★更新★：
   today + transfer_days[i,j] + local_delivery[j] <= confirm_date[j] + x[j]
   （调拨到达+本地配送 须在 目标终端客户确认日+x天 之内）

5. 最低调拨量约束：
   t[i,j] >= min_transfer_qty × z[i,j] 或 t[i,j] = 0

6. 终端装卸能力约束：
   Σ_j t[i,j] + r[i] <= daily_handling_capacity[i] × available_days
   Σ_i t[i,j] <= daily_handling_capacity[j] × available_days

7. 保质期约束(如适用)：
   remaining_shelf_life[i] - transfer_days[i,j] >= min_shelf_days

8. 回程运力约束：
   b[i,j] <= backhaul_available[i,j]

9. 非负约束：
   t[i,j], r[i], s[i] >= 0
```

### 4.4 ETA预测优化

```
输入特征：
  - 起运地、目的地
  - 出发月份（季节因子）
  - 承运商ID（承运商因子）
  - 历史ETA/ATA偏差统计
  
模型：
  - 初期：加权移动平均（历史ATA数据的加权均值）
  - 后期：梯度提升回归（XGBoost），以ATA为标签训练

输出：
  - 预测ETA
  - 置信区间(如95%置信区间)
  
反馈循环：
  每次实际到达(ATA)记录后，自动更新模型参数，缩小ETA与ATA的偏差
  ETA准确率提升 → 补货提前期L更精准 → 安全库存计算更准确
```

### 4.5 预测置信度更新机制 ★新增★

```
数据采集：
  每次订单确认后，记录到forecast_deviation_log表
  按4个维度分层：客户 × 目的地 × 时段 × 订单规模

计算方法(EWMA)：
  confidence_new = α × recent_accuracy + (1-α) × confidence_old
  α = 0.3（近期权重30%，历史惯性70%）
  recent_accuracy = 近3个月的预测准确率

偏差方向识别：
  若客户连续N次偏差方向一致(如总是多报)，则识别为系统性偏差
  自动计算偏差修正系数 = 实际确认量的均值 / 预测量的均值
  
更新频率：
  每周自动更新一次系统建议值
  管理员收到通知后确认或覆盖

初始化策略(Demo)：
  无历史数据时，管理员根据经验手动设置初始置信度
  系统后台从第一天开始积累偏差数据
  积累≥30个样本后，系统开始给出建议值
```

### 5.1 主流程（按时间线）

```
Phase 1: 预测接收与预置计划
  ① 累积接收销售预测（按交付日期和目的地分类汇总）
  ② 置信度修正预测量（预测 × 偏差修正系数）
  ③ 计算各终端安全库存水位（Z × σ × √L）
  ④ 确定预置深度（基于x值和在途时间）

Phase 2: 发货计划优化与执行
  ⑤ 引擎一计算最优分批滚动发货计划
  ⑥ 结果透明展示（成本明细、方案对比、x敏感性分析）
  ⑦ 用户确认执行

  ⑨ 运力不足 → 重新优化
  ⑩ 生成运单，锁定发货计划
  ⑪ 新到的销售预测归入下一批次

Phase 3: 在途跟踪与反馈
  ⑫ 记录实际发货时间
  ⑬ 基于承运商信息预测ETA
  ⑭ 记录ATA，更新ETA偏差统计 → 更新在途时间参数

Phase 4: 终端到货与订单确认（两条独立并行时间线） ★修正v3.2★
  ── 时间线A：货物侧 ──
  ⑮ 货物到达终端，入库，进入预置库存
  ⑮b 到货即检查：该订单是否已被客户确认？
     ├─ 已确认(customer_confirmed) → 直接进入汇合流程
     └─ 未确认(pending) → 立即发起ALARM给系统人员
  
  ── 时间线B：客户侧（与时间线A无先后依赖） ──
  ⑯ 客户根据自身需求提交订单确认（时间不定）
     ├─ 可能早于货物到达（提前确认）→ 到货时系统检查已确认，无告警
     ├─ 可能同步于货物到达
     ├─ 可能晚于货物到达（延迟确认）→ 到货时触发告警，客户后续自行确认则告警解除
     └─ 到货时未确认 → 系统人员收到ALARM后人工确认
  
  ── 汇合点：当两条时间线都就绪时 ──
  ⑰ 记录预测偏差(confirmed_qty vs forecast_qty) → 更新置信度和安全库存σ
  ⑱ 对比订单确认与销售预测差异

  ㉑ 调拨方案展示 + 用户确认
  ㉒ 滞留库存量反馈至引擎一下一批次
  ㉓ 反馈至客户，建议调整生产计划
```

### 5.3 运力匹配流程

```
1. 计算本批次各目的地需要的车辆数量和类型
2. 查询可用车辆资源池
3. 按[成本最低 → 装载率最高 → 承运商可靠性]优先级匹配
4. 输出运力匹配率 = 可承运托盘数 / 需承运托盘数 × 100%

   b. 提供备选方案：

      - 接受部分延迟（计算违约成本）

   c. 用户选择方案后重新优化
```

**告警触发条件★修正-到货即检查，非超时★**：
```
# 在货物到达终端的那一刻执行（ATA记录触发）
IF 货物已到达(ATA已记录)
   AND 该forecast_id对应的订单确认 status == "pending"
THEN 立即发起ALARM告警
     order.status = "arrival_alarm"
     order.alarm_triggered_at = ATA时间
```

**告警流程**：
```
① 系统发起告警
   - 在仪表板和待处理队列中显示标红提示
   - 发送邮件/短信通知指定的系统人员
   
② 系统人员查看告警详情
   - 货物信息：到达时间、预置量、所在终端
   - 预测信息：原始预测值、置信度、修正后建议值
   - 客户信息：客户联系方式、历史交付记录
   
③ 系统人员执行确认
   方式A：主动联系客户
   - 电话/微信/邮件联系客户销售部
   - 获取最新的订单需求确认
   - 若客户确认，则按客户确认值执行
   
   方式B：根据业务判断确认
   - 如果无法联系到客户，系统人员根据：
     · 该客户历史订单模式(往往确认值接近预测值)
     · 该终端当前库存和存储成本压力
     · 终端其他订单的紧急程度
     → 做出人工确认决策(通常采用修正后的建议值)
   
④ 在系统中记录人工确认
   输入以下信息：
   - 确认数量：[输入确认值]
   - 确认来源：[下拉选择：口头确认/邮件确认/电话确认/系统建议]
   - 备注说明：[输入文字，如"客户电话确认，需求100托，较预测少50托"]
   
   系统自动填充：
   - confirmed_by = "SYS_USER_xxx" (系统人员ID)
   - status = "manual_confirmed"
   - confirmed_at = now() (系统人员确认时的时间)
   
⑤ 确认完成后自动触发后续流程
   - 汇合检查：货物已到达 ✓ + 订单已确认 ✓
   - 进行差异分析(确认量 vs 预置量)
   - 触发本地配送(若确认量 > 0)
   - 触发引擎二调拨优化(若有剩余库存)
```

**关键设计要点**：
- **到货即检查**：不等待48小时或任何超时，货物一到达就立刻检查订单确认状态
- **人工确认权限**：不是所有系统人员都有权限确认，需要角色控制(如"订单确认管理员")
- **告警可解除**：如果系统人员处理告警前客户自行确认了，告警自动解除，status变为"customer_confirmed"
- **审计追踪**：每笔人工确认都被完整记录(谁、何时、确认了多少、依据是什么)，便于后续分析
- **与引擎一反馈**：人工确认的结果也计入预测偏差统计，用于更新置信度和安全库存

## 九、关键风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 优化问题规模增长后求解速度慢 | 生产环境不可接受 | Demo用OR-Tools，生产切Gurobi；加入启发式预处理 |
| 模拟数据与真实业务差异大 | 优化效果不可信 | 尽早拿到客户一年订单数据做回测 |
| 参数维护工作量大 | 用户体验差 | 提供合理默认值+批量导入功能 |
| 多目标冲突（成本 vs 时效） | 用户难以决策 | 统一换算为成本+方案对比+x敏感性滑块 |
| 装载率计算精度不足 | 实际和理论不符 | 预留5-10%安全裕度 |
| 预测置信度不准 ★新增★ | 预置库存过多或不足 | 人工初始化兜底+系统建议+上下限约束 |
| x=1预置深度大,库存风险高 ★新增★ | 库存积压成本高 | 分批滚动补货+实时监控+紧急补货通道 |
